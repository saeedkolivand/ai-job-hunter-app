The Agent Fleet

a swarm of tiny specialists builds this repo, and a second swarm tears their work apart.

24 agents, paired author + critic per domain.

Every task and code change in this repo routes through a .claude/ agent system. You name no agent — the system reads the files you touched, picks the author that owns that area, runs an independent critic over the diff, pulls in a security or performance secondary when there's risk, writes tests, sweeps dead code, and lets a steward close the docs. Nobody approves their own work. Click around below to see it move.

One Task, End to End

how a single change actually flows through the fleet.

1

Intake & triage main session

The diff's touched paths are matched against .claude/review-routes.json — first matching glob wins. That picks the Primary Owner (the author) and any risk Secondary. The main session orchestrates; agents never call other agents.

2

Pre-harvest handoff context, once

Instead of every agent cold-exploring the repo, the owning area is harvested once into a compact handoff file (symbols, paths, contracts). The author and its critic read that, not the whole tree — this is what keeps the multi-agent token premium small.

3

Author implements write access

The domain author (e.g. pdf-docx-generator) makes the smallest diff that satisfies the spec, then hands off. It does not approve itself.

4

Independent critic audits read-only

The paired critic (resume-export-expert) reviews the diff against its checklist. HIGH/CRITICAL findings block; LOW/MEDIUM are advisory. A fresh pair of eyes — never the writer.

5

test-author writes tests if testable

If the change touches testable logic, test-author adds coverage (unit / golden / e2e). No feature is complete without it.

6

testing-reviewer audits the tests read-only

testing-reviewer challenges weak assertions, flakiness, and untested error/security paths — and it never writes tests, so the author of the tests isn't the one signing off.

7

cleanup sweeps dead code report-first

cleanup audits for unused files / exports / deps the change orphaned, and deletes only the safe tier after confirmation.

8

project-steward closes sole doc writer

project-steward syncs docs/ + docs/knowledge, persists any lesson, and runs graphify update . + codegraph sync so the graphs match reality again.

9

pr-reviewer gate before the PR

The final internal pass before git push. pr-reviewer runs the repo's real tools (typecheck, lint, clippy, tests) + cross-file blast-radius + a verification gate so CodeRabbit finds less. 🔴+🟠 findings block the PR. Invoke with /review.

With vs Without the Fleet

same job. honestly: agents cost a token premium for higher reliability + parallel speed.

One agent, no fleet

Wall-clock Serial. One context does intake, code, review, tests, docs — back to back.
Tokens Cold re-exploration every spawn: ≈70–122k tokens/spawn, ≈120 tool-uses to rebuild context it never kept.
Bugs caught before release The writer reviews its own work — the missed error is the one it didn't see the first time. No release gate.

The agent fleet

Wall-clock Authors run in parallel on disjoint files; critics audit alongside. Faster on cross-layer work.
Tokens Pre-harvested handoff instead of cold start: ≈44 tool-uses vs ≈120. Multi-agent adds ≈+58% token overhead — kept small by the handoff.
Bugs caught before release An independent writer/critic split catches ≈60–80% of otherwise-missed errors, plus the Stop review-gate before merge.

ILLUSTRATIVE FIGURES. The numbers above (cold-start ≈70–122k tokens/spawn; ≈44 vs ≈120 tool-uses pre-harvested vs cold; writer/critic catching ≈60–80% of missed errors; multi-agent ≈+58% token overhead) are illustrative, not measured benchmarks of this repo. The honest trade: agents cost a token premium for higher reliability and parallel speed; the pre-harvested handoff is what keeps that premium small.
